# 今後の整理・追加 - 詳細計画

> **目的:** tasks.md と docs/plans の運用を整理し、今後の追加・改善の指針を明確にする。

**Goal:** タスク管理と計画システムの運用ルールを確立し、新規機能追加・メンテナンスのフローを文書化する。

**Architecture:** tasks.md に具体的なバックログと運用ルールを追加。docs/plans に計画ライフサイクルと追加候補を定義。ドキュメント間の整合性維持フローを確立する。

**Tech Stack:** Markdown, Chezmoi, GitHub Actions

**Reference:** [docs/tasks.md](../tasks.md), [docs/plan.md](../plan.md), [docs/requirements.md](../requirements.md)

---

## 1. 現状分析

### tasks.md の課題

| 課題                 | 内容                                                                   |
| -------------------- | ---------------------------------------------------------------------- |
| 汎用テンプレートのみ | 開発・メンテナンス・テストのカテゴリはあるが、具体的なバックログがない |
| 進捗の実体がない     | ステータス定義はあるが、実際のタスクと紐づいていない                   |
| 計画との連携が弱い   | plan.md への参照はあるが、どの計画がどのタスクに対応するか不明瞭       |

### docs/plans の課題

| 課題                 | 内容                                                                         |
| -------------------- | ---------------------------------------------------------------------------- |
| 計画が2件のみ        | WSL2 と plan-system-setup のみ。将来対応予定（Windows, FreeBSD）の計画がない |
| ライフサイクル未定義 | 計画の作成・完了・アーカイブのルールがない                                   |
| 定期レビューなし     | 計画の優先度変更や見直しのタイミングが不明                                   |

### ドキュメント間の整合性

| 関係                | 現状                                               |
| ------------------- | -------------------------------------------------- |
| requirements ↔ plan | 対応 OS のステータスが plan の一覧と乖離する可能性 |
| tasks ↔ plan        | 計画完了時に tasks を更新するルールがない          |
| problems ↔ 新規設定 | 新規追加時の problems 追記がチェックリストのみ     |

## 2. 改善方針

### 2.1 tasks.md の整理

**追加するセクション:**

1. **運用ルール** - タスク追加・完了時の更新手順
2. **具体的バックログ** - 優先度付きの実タスク一覧（plan と紐づけ）
3. **定期レビュー** - 月次/四半期の見直し項目

### 2.2 docs/plans の運用

**追加するルール:**

1. **計画ライフサイクル** - 作成 → 進行中 → 完了 → アーカイブ
2. **計画追加のトリガー** - 新規 OS 対応、大規模リファクタ、新ツール統合
3. **plan.md 一覧の更新** - 計画追加・完了時に必ず更新

### 2.3 ドキュメント更新フロー

```txt
新規機能/修正
    ↓
計画が必要？ → Yes → docs/plans/YYYY-MM-DD-<feature>.md 作成
    ↓                    ↓
    No                   plan.md 一覧に追加
    ↓                    ↓
tasks.md にタスク追加 ← 計画のタスクを tasks に反映（オプション）
    ↓
実装
    ↓
完了時: requirements.md, README, plan.md, tasks.md を更新
```

## 4. tasks.md への追加内容（案）

### 4.1 運用ルール

```markdown
## 運用ルール

### タスク追加時

1. 計画が必要な規模（新 OS 対応、大規模変更）→ `docs/plans/YYYY-MM-DD-<feature>.md` を作成
2. 計画不要な小規模タスク → 本セクションのバックログに直接追加
3. `docs/plan.md` の既存計画一覧に新規計画を追加

### タスク完了時

1. 該当計画があればステータスを「完了」に更新
2. `docs/requirements.md` の対応 OS ステータスを更新
3. `README.md` の「予定」「対応済み」を更新
4. 本ドキュメントのバックログから該当タスクを削除または完了マーク
```

## 5. plan.md への追加内容（案）

### 5.1 計画ライフサイクル

```markdown
## 計画ライフサイクル

| ステータス | 説明                             |
| ---------- | -------------------------------- |
| 未作成     | 計画ファイルが存在しない         |
| 未着手     | 計画あり、実装未開始             |
| 進行中     | 実装中                           |
| 完了       | 検証済み、ドキュメント更新済み   |
| 保留       | 一時停止（理由を計画内に記載）   |
| 中止       | 実施しない（理由を計画内に記載） |
```

### 5.2 計画追加のトリガー

- 新規 OS 対応（Windows, FreeBSD, WSL2 等）
- 大規模リファクタ（ディレクトリ構成変更、テンプレート統合等）
- 新ツールの統合（mise 以外のパッケージマネージャー等）
- セキュリティ関連の大幅変更
